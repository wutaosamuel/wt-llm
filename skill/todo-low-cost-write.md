# low-cost-write 成本分析与待办

## 背景问题

工作流对比：

- **方案 A（low-cost-write 技能）**：贵模型（Opus）生成完整内容并区分 old/new
  → 便宜模型（Sonnet）机械写入文件。
- **方案 B（直接编辑）**：贵模型（Opus）不生成完整替换内容，直接调用 Edit 改文件。
- **三段式**：Opus 决策/生成 → Sonnet 写入 → Opus 检查。

核心疑问：这样切换模型一般情况下能降低成本吗？

## 成本结构的关键事实

1. **Edit 工具的 old/new 参数本身就是模型的 output token。** 写入动作的成本
   = 产出这段 old/new 的 output 成本，而不是"写文件"这个物理动作。
2. **省钱的前提是让便宜模型承担推理/生成工作。** 如果内容和 old/new 判断全部
   由 Opus 完成，Sonnet 只做复制，就省不下 Opus 的任何 output。
3. **同一段内容会被计费多次**（方案 A）：
   - Opus output（生成）
   - Sonnet input（读入）+ Sonnet output（写入）
   相比方案 B 只付一次 Opus output，多了 Sonnet 的两笔。
4. **上下文历史（conversation history）是长会话的成本大头。** 每多切一次模型，
   若不清理上下文，就多重放一次历史 input。

## 方案 A 相对方案 B 的净效果

- 省下：写入 output 从贵单价 → 便宜单价
  ≈ `内容长度 × (贵 output 单价 − 便宜 output 单价)`
- 新增：`(内容长度 + SKILL 加载 + 验证读回) × 便宜模型单价`
- **净效果 = 省的 − 新增。**

结论：

- 内容**越长** → 省的部分线性增长，固定开销占比变小 → **越划算**。
- 内容**很短** → SKILL 加载 + 验证读回等固定开销可能反超 → **反而更贵**。

## 三段式（Opus → Sonnet → Opus 检查）专项结论

**一般情况下不能降低成本，通常反而更贵**，原因：

1. Opus 已生成全部内容，Sonnet 没省下任何思考，只是把内容再读入 + 再吐出，
   多付 Sonnet 两笔。
2. **第三步"Opus 检查"是成本重灾区**：Opus 需读回文件（input）+ 原始内容留在
   上下文（input）+ 输出判断（output）。仅这一步的 input 开销就可能超过
   "Opus 直接编辑"整个动作的成本。
3. 若三步同处一条对话链，历史上下文被反复重放，进一步推高成本。

## 何时该用 low-cost-write（必须同时满足）

1. 写入内容**极长**（数百至上千行），使写入 output 的单价差足够大；
2. Sonnet 在**干净最小上下文**中运行——只带 SKILL + 内容，不背负 Opus 历史；
3. **去掉第三步的 Opus 检查**，改由 Sonnet 自己读回自验证
   （SKILL.md 规则 6/7 已如此设计）。

只要保留"Opus 检查"，第三步的 input 成本基本会吃掉甚至反超前面省下的钱。

## 直觉对照表

| 场景 | 比"Opus 直接编辑"更便宜？ |
|---|---|
| 短内容（几十行内） | 否，更贵（固定开销反超） |
| 长内容 + Sonnet 干净上下文 + Sonnet 自验证（无 Opus 检查） | 是，可能便宜 |
| 长内容 + 保留 Opus 检查 | 否，通常更贵（Opus 检查 input 太贵） |
| 三步都在同一长对话链里 | 否，更贵（历史被反复重放） |

## 推荐形态

> Opus 生成内容 → **丢弃 Opus 上下文** → Sonnet 用最小上下文写入
> → **Sonnet 自己读回验证**（不回到 Opus）

即把检查权交给 Sonnet，Opus 一旦交棒不再回来。
